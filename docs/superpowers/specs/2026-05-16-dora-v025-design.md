## 2. Multi-Table Tabs

### Goal
Allow users to have multiple tables open simultaneously, removing the current single-table constraint.

### Architecture

**Tab store:** `src/core/tabs/tab-store.ts` — zustand store.

```ts
type Tab = {
  id: string          // uuid
  connectionId: string
  tableId: string
  tableName: string
  label: string       // display name, usually tableName
}

type TabStore = {
  tabs: Tab[]
  activeTabId: string | null
  openTab: (tab: Omit<Tab, 'id'>) => void   // opens or focuses existing tab
  closeTab: (id: string) => void
  setActiveTab: (id: string) => void
}
```

- `openTab` checks if a tab for `(connectionId, tableId)` already exists; if so, focuses it rather than duplicating.
- Max 12 tabs. Opening a 13th replaces the least-recently-used tab.

**Tab bar:** New `src/features/tab-bar/tab-bar.tsx` component. Rendered in `Index.tsx` between the sidebar and the main content pane. Horizontal strip of tab pills with:
- Table name as label
- Small DB provider icon (reuses existing provider icons)
- × close button per tab
- Horizontal scroll when > 8 tabs visible
- Middle-click to close (via `onAuxClick`)
- Active tab highlighted

**DatabaseStudio isolation:** `Index.tsx` renders `<DatabaseStudio key={tab.id} ... />` for the active tab. The React `key` ensures each tab gets a fully isolated component tree — pagination, filters, sort, scroll position, and selection are all per-tab. Unmounted tabs are not kept in memory; switching back re-fetches data.

**Sidebar integration:** Clicking a table in `DatabaseSidebar` calls `tabStore.openTab(...)` instead of directly setting `selectedTableId`. The sidebar no longer manages selected table state — the tab store owns it.

**Empty state:** When `tabs.length === 0`, renders the existing "No table selected" empty state.

### Data flow

```
Sidebar click → tabStore.openTab()
                     ↓
            Tab bar re-renders (new pill)
                     ↓
         Index.tsx renders DatabaseStudio[key=tab.id]
                     ↓
              loadTableData() fires for that tab
```

### Error handling
If a connection is disconnected while a tab is open, the tab pill shows a red dot indicator. The `DatabaseStudio` instance shows its existing connection-error state.

## 3. FK Drill-Down Navigation

### Goal
Let users click a foreign-key cell value to instantly open the referenced row in a new tab.

### Data

`ColumnDefinition` (currently `{ name, type, nullable, primaryKey }`) gains an optional field:

```ts
foreignKey?: {
  referencedTable: string
  referencedColumn: string
  referencedSchema?: string
}
```

**Backend:** FK metadata is already fetched for the schema visualizer from `information_schema.key_column_usage`. The `read` adapter's `fetchTableData` is extended to join this data into the column list. Only the Postgres and MySQL adapters need this (SQLite has `PRAGMA foreign_key_list`, LibSQL same).

**No new Tauri commands.** The existing `fetch_table_data` command's response shape grows one optional field per column.

### UI

In `data-grid.tsx`:

- On row hover, FK columns render a small ↗ icon (12px, muted color) to the right of the cell value.
- Clicking the ↗ icon (not the cell text) triggers FK navigation. This avoids conflict with double-click to edit.
- Non-FK columns are unaffected.

### Navigation

```ts
// Icon click handler
tabStore.openTab({
  connectionId: activeConnectionId,
  tableId: col.foreignKey.referencedTable,
  tableName: col.foreignKey.referencedTable,
  label: col.foreignKey.referencedTable,
})

// Then apply a filter to show the referenced row:
// { column: col.foreignKey.referencedColumn, operator: '=', value: cellValue }
```

The tab opens with the filter pre-applied using the existing `FilterDescriptor` system, so the referenced row is immediately visible. If the referenced table is already open in a tab, that tab is focused and its filter is updated.

### Edge cases
- **Circular references:** Opening a tab that opens another tab that opens back is prevented by the "focus existing tab" behavior in `openTab`.
- **NULL FK values:** ↗ icon is not shown when the cell value is NULL.
- **Composite FKs:** Not supported in v0.25. Only single-column FKs show the icon.

## 4. CSV Import with Smart Column Mapping

### Goal
Allow users to import CSV data into an existing table with a preview and column-mapping step.

### Entry point
"Import CSV" button in `StudioToolbar`, grouped with existing export buttons. Only shown when a table is selected.

### Flow

**Step 1 — File pick**  
Native `<input type="file" accept=".csv,.tsv">`. File is read with the browser `FileReader` API and parsed client-side. No new npm dependency — a simple parser (~50 lines) handles standard RFC 4180 CSV including quoted fields and escaped quotes.

**Step 2 — Preview & mapping dialog**  
A modal dialog (`ImportCsvDialog`) with two panels:

- **Left:** CSV preview table showing the first 5 rows and all detected headers.
- **Right:** Column mapping list. Each CSV header maps to a DB column via a `<select>` dropdown. Auto-matched by exact name (case-insensitive) on open. Unmatched headers shown in amber with "Skip" as the default. Type mismatch warnings shown inline (e.g., "CSV has 'abc' but column is `integer`").

Additional options:
- "Skip first row" toggle (for files where row 1 is data, not headers).
- "Stop on first error" vs "Skip errors and continue" toggle (default: skip).

**Step 3 — Import**  
Iterates rows, calls existing `insert_row` Tauri command per row. The PK column is automatically excluded from the mapped columns.

A progress bar shows `Importing row N of M...`. Import runs in a loop (not parallelized) to respect DB connection limits.

**Step 4 — Result**  
On completion: toast with "Imported X rows" (or "Imported X rows, Y skipped"). If errors occurred, an expandable "View errors" section lists the row number and error message for each failure. Table data is reloaded automatically.

**Large file warning:** Files > 5000 rows show a warning: "This may take a moment."

### No new backend commands
All writes go through the existing `insert_row` command. The entire feature is frontend-only except for the Tauri call.

### Component structure

```
src/features/database-studio/components/
  import-csv-dialog.tsx      — main dialog
  import-csv-preview.tsx     — 5-row preview table
  import-csv-mapping.tsx     — column mapping UI
src/features/database-studio/utils/
  csv-parser.ts              — minimal RFC 4180 parser
  csv-type-check.ts          — value-vs-column-type validation
```
